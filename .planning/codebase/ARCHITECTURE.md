# Architecture

**Analysis Date:** 2026-06-08

## Pattern Overview

**Overall:** Microservices Architecture (Decentralized Database pattern)

**Key Characteristics:**
- **Service Registry & Discovery:** Uses Netflix Eureka Server to maintain a registry of active services and dynamic routing.
- **API Gateway:** Uses Netflix Zuul to route client requests to appropriate services, serving as a single entry point for clients.
- **Database-per-Service:** Each microservice (`user-service`, `product-catalog-service`, `product-recommendation-service`, `order-service`) has its own SQL Server database to ensure loose coupling.
- **Shared Session State:** Shares HTTP session state across microservices using Spring Session backed by Redis.
- **Synchronous Communication:** Relies on declarative REST clients (Spring Cloud OpenFeign) for inter-service RPC queries.

## Layers

Each business microservice (e.g., `user-service`, `product-catalog-service`) follows a typical layered architecture:

**Controller (Web/API) Layer:**
- Purpose: Exposes RESTful endpoints, handles HTTP requests/responses, and maps routes.
- Location: `src/main/java/com/rainbowforest/[service]/controller/`
- Depends on: Service Layer, HTTP/Header Layer.
- Used by: External clients via the API Gateway.

**Service (Business Logic) Layer:**
- Purpose: Implements domain logic, coordinates data access, and maps model conversions.
- Location: `src/main/java/com/rainbowforest/[service]/service/`
- Depends on: Repository Layer, Feign Clients.
- Used by: Controller Layer.

**Repository (Data Access) Layer:**
- Purpose: Manages raw data storage access using SQL Server (Spring Data JPA) or Redis (`RedisTemplate`).
- Location: `src/main/java/com/rainbowforest/[service]/repository/` or `.../redis/`
- Depends on: JDBC/JPA config, Redis connection.
- Used by: Service Layer.

**Entity / Domain Layer:**
- Purpose: Represents persistent database models or local mock models for remote data transfer.
- Location: `src/main/java/com/rainbowforest/[service]/entity/` or `.../domain/` or `.../model/`

**HTTP/Header Layer:**
- Purpose: Utility classes generating standard headers (e.g. location, content type) for success or error responses.
- Location: `src/main/java/com/rainbowforest/[service]/http/header/`

## Data Flow

**Placing an Order Flow:**
1. A client sends a POST request `/api/shop/order/{userId}` with a `Cookie` header (holding the cart session ID) to the API Gateway.
2. `api-gateway` routes the request to the `order-service` instance, forwarding the Session cookie.
3. `OrderController` in `order-service` receives the request and fetches the current shopping cart items from Redis via `CartService`.
4. `OrderController` calls `UserClient` (Feign Client) to query `user-service` at `/users/{userId}` to verify if the user exists.
5. Once user details are fetched, the controller constructs the `Order` object, calculates totals, and calls `OrderService` to save it to the `orders` SQL Server database.
6. The controller clears the shopping cart from Redis via `CartService`.
7. `order-service` returns the created `Order` object to the Gateway, which returns it to the client.

**State Management:**
- **Relational Data:** Persisted across SQL Server databases (tables for users, user details, products, recommendations, orders, and order items).
- **Session & Cart Data:** Cached and retrieved dynamically from Redis.

## Key Abstractions

**Repository Pattern:**
- Spring Data `JpaRepository` is used for CRUD operations (e.g., `UserRepository`, `ProductRepository`, `OrderRepository`).
- Custom repository wrapper `CartRedisRepository` is used for Redis hash operations.

**Feign Client Abstraction:**
- `@FeignClient` interfaces (e.g., `UserClient`, `ProductClient`) hide HTTP call mechanics to simplify inter-service queries.

**Header Generator:**
- `HeaderGenerator` class helps standardize REST headers for HTTP GET, POST, and error handling.

## Entry Points

**API Gateway:**
- Location: `api-gateway` (`ApiGatewayApplication.java`) at port `8900`.
- Responsibilities: Routes incoming public calls to target services and attaches cookie headers.

**Business Services:**
- Each microservice boots via its own Spring Boot class (`UserServiceApplication`, `ProductCatalogServiceApplication`, etc.).
- Eureka Discovery manages routing paths automatically.

## Error Handling

**Strategy:** Exception catching inside Controller request mappings.
- Try-Catch blocks surround database writes in controllers.
- Failure returns standard `ResponseEntity` with error headers generated by `HeaderGenerator` and HTTP status (`HttpStatus.INTERNAL_SERVER_ERROR`, `HttpStatus.BAD_REQUEST`).
